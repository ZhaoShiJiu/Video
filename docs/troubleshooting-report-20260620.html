🔍 排查时间线

T+0 min
🟡 用户报告：Docker WebUI 返回 HTTP 500
运行 docker-compose up -d 后，浏览器访问 127.0.0.1:8501 显示 Internal Server Error。 初步怀疑近期代码变更（tagging 标签库功能）引入了 config 读取异常。
T+5 min
🟠 代码审查：排除 config.tagging 引用问题
逐个审查了 Main.py、task.py、tagging.py 中所有 config.tagging.get() 调用 —— 全部使用了安全默认值 .get("enabled", True)，不会因 AttributeError 导致崩溃。 config.toml 中也确实存在 [tagging] 段。
结论：代码逻辑正确，问题不在这里。
T+8 min
🟠 防御性加固：修复 config.py 编码处理
发现 load_config() 只尝试 UTF-8 → utf-8-sig 两次回退，任意一次失败都会导致未处理异常。 增加 5 种编码自动回退（utf-8 / utf-8-sig / gbk / gb2312 / latin-1）， 全部失败时返回空配置 {} 兜底。
T+12 min
🔵 关键发现：容器根本没有运行！
docker ps -a 显示 WebUI 容器状态为 Created（不是 Up）。 docker start 手动启动时报出真正错误： 
Error: port 127.0.0.1:8501 → bind:
An attempt was made to access a socket
in a way forbidden by its access permissions.
T+15 min
🟢 根因定位：Windows 端口保留范围
netsh interface ipv4 show excludedportrange 显示： 端口 8501 落在 Windows 保留范围 8498–8597 内，Docker 无权绑定。 这解释了为什么 API 容器 (8080) 正常，而 WebUI (8501) 失败。
T+18 min
🟣 修复 & 验证
将 WebUI 端口从 8501 改为 8700（落在可用范围 8698–8797）， 重启容器后 Streamlit 正常启动，HTTP 200 响应确认。
🔴 根因
Windows 系统保留端口 8501 被排除在 Docker 可绑定范围之外

Windows Hyper-V / WSL 将端口段 8498–8597 列入 TCP 排除范围， Streamlit 默认端口 8501 恰好在其中。Docker 尝试绑定该端口时被操作系统拒绝， 容器创建成功但无法启动，导致浏览器访问时出现 HTTP 500。

📝 经验教训

#1 先看容器状态，再看应用日志 —— docker ps -a 比直接查 log 更优先。容器状态 Created ≠ Up，意味着端口绑定层面的问题，根本没到 Python 代码执行阶段。
#2 Windows + Docker 的端口陷阱 —— Windows 会为 Hyper-V/WSL 保留大量端口段（连续 100 端口为一块）。部署前用 netsh interface ipv4 show excludedportrange 检查目标端口是否可用。
#3 不要过早下结论 —— 最初怀疑是最近的 tagging 代码改动导致 config 读取异常，花了不少时间审查 Python 代码。实际根因在基础设施层（Windows 端口策略），与代码改动无关。
#4 防御性编码仍有价值 —— 虽然 config.py 编码问题不是本次根因，但加固后的多编码回退机制消除了一个潜在的单点故障，属于"顺手修"的收益。
